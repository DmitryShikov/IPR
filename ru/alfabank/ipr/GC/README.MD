# Практическая работа: Настройка Garbage Collector (G1 GC)

## Цель работы

Настроить **Garbage Collector G1** для высоконагруженного приложения, добившись:

* снижения длительных пауз (*long pauses*);
* уменьшения скорости роста **Old Generation**;
* более эффективного использования памяти Heap.

---

# 1. Описание тестовой программы

## Назначение

Программа имитирует нагрузку типичного микросервиса и создаёт три категории объектов:

| Тип объектов   | Доля | Особенности                                    |
| -------------- | ---- | ---------------------------------------------- |
| Короткоживущие | 70%  | Быстро создаются и удаляются                   |
| Средней жизни  | 20%  | Переживают несколько циклов GC                 |
| Долгоживущие   | 10%  | Сохраняются в кэше и попадают в Old Generation |

---

## Почему такая нагрузка создаёт проблемы для GC?

| Фактор                               | Влияние на GC                         |
| ------------------------------------ | ------------------------------------- |
| Предзаполнение кэша                  | Old Generation занят уже при старте   |
| Постоянное добавление данных         | Old Generation непрерывно растёт      |
| Удаление объектов из кэша            | Появляется фрагментация и мусор в Old |
| Массовое создание временных объектов | Давление на Eden и частые Young GC    |

---

# 2. Запуск с неудачной конфигурацией

```text
java -Xmx512m -Xms512m \
     -XX:+UseG1GC \
     -Xlog:gc*:file=gc_bad.log \
     LoadSimulator
```

## Параметры JVM

| Параметр         | Значение | Комментарий                    |
| ---------------- | -------- | ------------------------------ |
| Heap Size        | 512 MB   | Минимальный размер памяти      |
| Region Size      | 1 MB     | Размер региона G1 по умолчанию |
| MaxGCPauseMillis | 200 ms   | Целевая пауза GC               |
| G1NewSizePercent | 5%       | Небольшой размер Eden          |

---

# 3. Анализ «плохого» GC-лога

### Фрагмент лога

```text
[0.022s] GC(0) Old regions: 2->22
[0.022s] GC(0) Pause Young 26M->24M 3.5ms

[0.025s] GC(1) Old regions: 22->39
[0.025s] GC(1) Pause Young 45M->41M 2.5ms

[0.049s] GC(2) Old regions: 39->43
[0.049s] GC(2) Pause Young 64M->41M 0.8ms

[0.086s] GC(3) Old regions: 43->43
[0.086s] GC(3) Pause Young 85M->42M 0.6ms
```

---

## Выявленные проблемы

| Проблема                       | Проявление в логе                           | Критичность |
| ------------------------------ | ------------------------------------------- | ----------- |
| Быстрый рост Old Gen           | `2 → 22 → 39 → 43` регионов за 0.1 сек      | 🔴 Высокая  |
| Неполная очистка памяти        | После GC остаётся ~41 MB                    | 🔴 Высокая  |
| Маленький Survivor Space       | `Survivor regions: 0→4(4)`                  | 🟡 Средняя  |
| Ранняя промоция объектов в Old | Объекты не успевают удерживаться в Survivor | 🔴 Высокая  |

---

## Последствия

### Быстрое заполнение Old Generation

За первые доли секунды количество регионов увеличивается:

```text
2 → 22 → 39 → 43
```

### Возможные проблемы при длительной работе

* запуск **Mixed GC** с паузами 500–2000 мс;
* рост времени отклика приложения;
* достижение предела памяти Heap;
* запуск **Full GC** с паузами в несколько секунд.

### Влияние на производительность

* увеличение Response Time;
* деградация пропускной способности сервиса;
* нестабильная работа приложения под нагрузкой.

---

# 4. Оптимизация G1 GC

## Конфигурация JVM

```text
java -Xmx512m -Xms512m \
     -XX:+UseG1GC \
     -XX:G1HeapRegionSize=4m \
     -XX:+UnlockExperimentalVMOptions \
     -XX:G1NewSizePercent=15 \
     -XX:G1HeapWastePercent=10 \
     -XX:ConcGCThreads=2 \
     -Xlog:gc*:file=gc_optimized.log \
     ru.alfabank.ipr.GC.LoadSimulator
```

---

# 5. Что изменилось после настройки

## Увеличен размер региона

### Было

```text
G1HeapRegionSize = 1 MB
Количество регионов = 512
```

### Стало

```text
G1HeapRegionSize = 4 MB
Количество регионов = 128
```

### Эффект

* уменьшается количество регионов;
* снижаются накладные расходы на управление памятью;
* сокращается объём метаданных GC.

---

## Увеличен размер Eden

### Было

```text
G1NewSizePercent = 5%
≈ 25 MB Eden
```

### Стало

```text
G1NewSizePercent = 15%
≈ 77 MB Eden
```

### Эффект

* больше короткоживущих объектов удаляется в Young GC;
* реже выполняются сборки мусора;
* меньше объектов попадает в Old Generation.

---

## Увеличен порог мусора для Mixed GC

### Было

```text
G1HeapWastePercent = 5%
```

### Стало

```text
G1HeapWastePercent = 10%
```

### Эффект

* Mixed GC запускается реже;
* уменьшается количество дорогостоящих циклов очистки.

---

# 6. Анализ «хорошего» лога

### Фрагмент лога

```text
[0.083s] GC(0) Old regions: 1->9
[0.083s] GC(0) Pause Young 77M->41M 6.0ms

[0.128s] GC(1) Old regions: 9->11
[0.128s] GC(1) Pause Young 105M->41M 1.8ms
```

---

## Наблюдения

### После первого Young GC

```text
Old regions: 1 -> 9
```

Рост составляет:

```text
8 регионов × 4 MB = 32 MB
```

### После второго Young GC

```text
Old regions: 9 -> 11
```

Рост всего на:

```text
2 региона × 4 MB = 8 MB
```

Скорость заполнения Old Generation существенно снизилась.

---

# 7. Сравнение результатов

| Показатель               | До оптимизации | После оптимизации | Изменение            |
| ------------------------ | -------------- | ----------------- | -------------------- |
| Рост Old после GC(0)     | +20 регионов   | +8 регионов       | ↓ в 2.5 раза         |
| Рост Old после GC(1)     | +17 регионов   | +2 региона        | ↓ в 8.5 раза         |
| Общее число Old-регионов | 43             | 11                | ↓ почти в 4 раза     |
| Размер региона           | 1 MB           | 4 MB              | ↑ в 4 раза           |
| Размер Eden              | 25 MB          | 77 MB             | ↑ более чем в 3 раза |

---

# 8. Вывод

Выполненная настройка G1 GC позволила:

* значительно сократить скорость роста **Old Generation**;
* уменьшить количество объектов, преждевременно попадающих в Old;
* увеличить эффективность очистки Young Generation;
* снизить вероятность запуска дорогостоящих Mixed GC и Full GC;
* повысить стабильность работы приложения под высокой нагрузкой.

Наиболее эффективными настройками оказались:

```text
-XX:G1HeapRegionSize=4m
-XX:G1NewSizePercent=15
-XX:G1HeapWastePercent=10
```

В результате удалось добиться более предсказуемой работы сборщика мусора и улучшить общую производительность приложения.
